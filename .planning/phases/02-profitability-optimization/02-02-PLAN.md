<objective>
Build an intelligent suggestion engine that analyzes blocker diagnostics data, maps high-frequency blockers to configurable thresholds, validates adjustments via backtesting, and surfaces the top 3 improvement suggestions with one-click apply.

Purpose: Blocker frequency data reveals which entry filters reject the most trades, but high rejection alone does not mean the threshold should change. By backtest-validating each potential adjustment, we ensure suggestions actually improve profitability before surfacing them. This closes the loop from analytics to action.

Output: Suggestion service (PROF-04), API endpoints for suggestion generation/apply/tracking, UI cards in Analytics tab with before/after metrics, apply buttons, and post-apply tracking.
</objective>

    **1. Create `src/services/suggestionService.js`:**

    This service connects blocker diagnostics to backtest-validated threshold suggestions.

    **a. Define BLOCKER_THRESHOLD_MAP** — Maps normalized blocker patterns to config keys and relaxation parameters:

    ```javascript
    export const BLOCKER_THRESHOLD_MAP = {
      'Prob X < X':           { configKey: 'minProbMid',         direction: 'lower',  step: 0.01,  minBound: 0.50,  label: 'Min Probability (Mid)' },
      'Edge X < X':           { configKey: 'edgeMid',            direction: 'lower',  step: 0.005, minBound: 0.005, label: 'Edge Threshold (Mid)' },
      'RSI in no-trade band': { configKey: 'noTradeRsiMin',      direction: 'lower',  step: 5,     minBound: 15,    label: 'RSI No-Trade Min' },
      'Entry price too high': { configKey: 'maxEntryPolyPrice',  direction: 'higher', step: 0.001, maxBound: 0.015, label: 'Max Entry Poly Price' },
      'Low liquidity':        { configKey: 'minLiquidity',       direction: 'lower',  step: 100,   minBound: 0,     label: 'Min Liquidity' },
      'High spread':          { configKey: 'maxSpreadThreshold', direction: 'higher', step: 0.002, maxBound: 0.030, label: 'Max Spread Threshold' },
      'Low impulse':          { configKey: 'minSpotImpulse',     direction: 'lower',  step: 0.0001, minBound: 0,    label: 'Min Spot Impulse' },
      'Choppy':               { configKey: 'minRangePct20',      direction: 'lower',  step: 0.0002, minBound: 0,    label: 'Min Range Pct (20)' },
      'Low conviction':       { configKey: 'minModelMaxProb',    direction: 'lower',  step: 0.01,  minBound: 0.50,  label: 'Min Model Max Prob' },
    };
    ```

    Note: blocker keys are already normalized by TradingState._normalizeBlockerKey() which replaces decimals with X and integers with N. The patterns above match those normalized forms.

    **b. Implement `matchBlockerToThreshold(blockerKey)`:**

    Iterate BLOCKER_THRESHOLD_MAP entries. For each pattern, check if the blockerKey starts with (or matches) the pattern. Return the matching entry or null. Use a prefix match since normalized blocker keys may have trailing content (e.g., "Prob X < X" matches "Prob X < X (inferred)" etc.).

    **c. Implement `computeRelaxedValue(currentValue, mapping)`:**

    Based on `mapping.direction`:
    - `'lower'`: return `Math.max(currentValue - mapping.step, mapping.minBound)`
    - `'higher'`: return `Math.min(currentValue + mapping.step, mapping.maxBound)`

    Return null if already at the bound (no further relaxation possible).

    **d. Implement `generateSuggestions(trades, blockerSummary, currentConfig, baseConfig)`:**

    Main function. Parameters:
    - `trades`: historical trades from ledger
    - `blockerSummary`: `{ total, topBlockers: [{ blocker, count, pct }] }` from TradingState
    - `currentConfig`: current engine config values (threshold keys)
    - `baseConfig`: base config for backtester comparison

    Algorithm:
    1. Filter `blockerSummary.topBlockers` to those matching BLOCKER_THRESHOLD_MAP
    2. For each matched blocker (sorted by frequency):
       a. Get current threshold value from `currentConfig[mapping.configKey]`
       b. Compute relaxed value via `computeRelaxedValue()`
       c. If relaxed value is null (at bound), skip
       d. Run backtest with current config (baseline) via `replayTrades(trades, {}, baseConfig)`
       e. Run backtest with relaxed value via `replayTrades(trades, { [mapping.configKey]: relaxedValue }, baseConfig)`
       f. Compare: only include if relaxed PF > baseline PF
       g. Determine confidence: green (50+ trades in relaxed), yellow (30-49), red (<30)
       h. Build suggestion object
    3. Sort by PF improvement descending
    4. Return top 3 suggestions

    Import `replayTrades` from `../domain/backtester.js`.

    Each suggestion object shape:
    ```javascript
    {
      configKey: string,
      label: string,
      currentValue: number,
      suggestedValue: number,
      blockerKey: string,
      blockerFrequency: number,  // pct
      baseline: { winRate, profitFactor, tradeCount },
      projected: { winRate, profitFactor, tradeCount },
      pfImprovement: number,  // projected PF - baseline PF
      confidence: 'green' | 'yellow' | 'red',
    }
    ```

    **e. Implement `computeConfidence(tradeCount)`:**

    - tradeCount >= 50: return 'green'
    - tradeCount >= 30: return 'yellow'
    - else: return 'red'

    **2. Create `test/suggestionService.test.js`:**

    - Test `matchBlockerToThreshold`: known blocker keys match correct config keys
    - Test `matchBlockerToThreshold`: non-threshold blockers (e.g., "Trading disabled") return null
    - Test `computeRelaxedValue`: lower direction reduces value by step
    - Test `computeRelaxedValue`: at minBound returns null
    - Test `computeConfidence`: 50+ -> green, 35 -> yellow, 10 -> red
    - Test `generateSuggestions` with mock data:
      - Create fake trades with known PnL
      - Create fake blockerSummary with high-frequency threshold blocker
      - Verify suggestions are returned with correct shape
      - Verify only PF-improving suggestions are included
      - Verify max 3 suggestions returned
      - Verify sorted by PF improvement descending
  </action>

    **1. Add `GET /api/suggestions` endpoint:**

    - Requires engine to be initialized
    - Load trades from paper ledger (reuse existing pattern from /api/analytics)
    - Get blocker summary from `engine.state.getBlockerSummary(25)`
    - Build currentConfig object (reuse pattern from /api/config/current)
    - Build baseConfig object (reuse pattern from /api/optimizer)
    - Call `generateSuggestions(trades, blockerSummary, currentConfig, baseConfig)`
    - Include `lastSuggestionTradeCount` and `tradesSinceLastAnalysis` in response
    - Track: store closed trade count at suggestion generation time in `globalThis.__lastSuggestionTradeCount`
    - On each call, compute `tradesSinceLastAnalysis` = current closed count - lastSuggestionTradeCount
    - Also check: if `totalEntryChecks < 100`, return empty suggestions with `{ insufficient: true, message: 'Need more blocker data (at least 100 entry checks)' }`

    Import `generateSuggestions` from `../services/suggestionService.js`.

    **2. Add `POST /api/suggestions/apply` endpoint:**

    - Request body: `{ configKey, suggestedValue, projected: { winRate, profitFactor, tradeCount } }`
    - Validate configKey is in OPTIMIZER_ALLOWED_KEYS or extend the allowed set
    - Apply the config change using the same pattern as POST /api/config (store previous, apply)
    - Record tracking data in `globalThis.__appliedSuggestions` array:
      ```javascript
      { configKey, suggestedValue, projectedWR, projectedPF, appliedAt: Date.now(), tradeCountAtApply: closedTradeCount }
      ```
    - Return success with applied value

    **3. Add `GET /api/suggestions/tracking` endpoint:**

    - Read `globalThis.__appliedSuggestions` array
    - For each applied suggestion:
      - Count trades closed after `appliedAt`
      - Compute actual WR and PF from those trades
      - Compare actual vs projected
      - Flag as 'underperforming' if actual PF < projected PF * 0.7
    - Return array of tracking records with actual vs projected metrics

    **Important:** The tracking endpoint needs access to trades. Load from paper ledger, filter to trades with exitTime after appliedAt timestamp.
