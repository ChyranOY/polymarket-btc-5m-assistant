<feature>
  <name>Order Lifecycle State Machine</name>
  <files>src/domain/orderLifecycle.js, test/domain/orderLifecycle.test.js</files>
  <behavior>
    OrderLifecycle class tracks order state transitions with timestamps.

    Valid states: SUBMITTED, PENDING, FILLED, PARTIAL_FILL, MONITORING, EXITED, TIMED_OUT, CANCELLED, FAILED
    Valid transitions:
      SUBMITTED -> PENDING, TIMED_OUT, FAILED, CANCELLED
      PENDING -> FILLED, PARTIAL_FILL, TIMED_OUT, FAILED, CANCELLED
      FILLED -> MONITORING
      PARTIAL_FILL -> MONITORING
      MONITORING -> EXITED, FAILED
      Terminal states: EXITED, TIMED_OUT, CANCELLED, FAILED (no transitions out)

    Cases:
      new OrderLifecycle('ord1', {tokenID: 'tok1'}) -> state='SUBMITTED', timestamps.SUBMITTED set
      lifecycle.transition('PENDING') -> true, state='PENDING', timestamps.PENDING set
      lifecycle.transition('INVALID') -> false, state unchanged
      lifecycle.isTimedOut(30000) at SUBMITTED + 31s -> true
      lifecycle.isTimedOut(30000) at SUBMITTED + 10s -> false
      lifecycle.isTerminal() at EXITED -> true
      lifecycle.isTerminal() at MONITORING -> false
      lifecycle.recordFill(50, 0.55) -> fillSize=50, fillPrice=0.55
      lifecycle.recordPartialFill(30, 0.55, 50) -> fillSize=30, requestedSize=50, fillRatio=0.6
  </behavior>
  <implementation>
    Pure class with no I/O. State stored as string, transitions validated against TRANSITIONS map.
    Each transition records timestamp. Terminal states prevent further transitions.
    fillSize/fillPrice tracked for partial fill ratio logging.
  </implementation>
</feature>

<feature>
  <name>Retry Policy and Error Classification</name>
  <files>src/domain/retryPolicy.js, test/domain/retryPolicy.test.js</files>
  <behavior>
    isRetryableError(err) classifies errors:
      {code: 'ECONNRESET'} -> true
      {code: 'ETIMEDOUT'} -> true
      {code: 'ENOTFOUND'} -> true
      {response: {status: 500}} -> true
      {response: {status: 502}} -> true
      {response: {status: 429}} -> true
      {response: {status: 401}} -> false (fatal)
      {response: {status: 403}} -> false (fatal)
      {response: {status: 400}} -> false (fatal)
      {response: {status: 422}} -> false (fatal)
      {message: 'fetch failed'} -> true (network error)

    withOrderRetry(fn, opts) executes fn with retry:
      fn succeeds first try -> returns result, no delay
      fn fails retryable 2x then succeeds -> returns result after 1s + 2s delays
      fn fails retryable 3x -> throws after 1s + 2s + 4s delays
      fn fails fatal first try -> throws immediately, no retry
      opts.maxAttempts=3, opts.delays=[1000,2000,4000], opts.maxDelayMs=30000

    createFailureEvent(orderId, error, retryCount) creates structured event:
      -> { type: 'ORDER_FAILED', orderId, error: {message, code, status, retryable}, retryCount, timestamp, severity: 'critical', category: 'order_execution' }
  </behavior>
  <implementation>
    Pure functions with no I/O except the async sleep in withOrderRetry.
    Error classification checks err.code first (Node.js network errors), then err.response.status (HTTP errors), then err.message patterns.
    withOrderRetry uses for-loop with try/catch, sleeping between attempts.
    createFailureEvent returns a plain object matching the schema from CONTEXT.md.
  </implementation>
</feature>
